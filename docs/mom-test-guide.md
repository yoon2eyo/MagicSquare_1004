# Mom Test 가이드 — 요구 구체화 · C2C Concept

> **대상:** 학습자 · 코딩 에이전트  
> **역할:** C2C **Concept** 단계 — 솔루션 전에 **진짜 문제**를 뽑는다  
> **연결:** [`PRD.md`](PRD.md) INV-1 · [`c2c-dual-track-concepts.md`](c2c-dual-track-concepts.md) · [`learning-guide.md`](learning-guide.md) §①

---

## 한 줄 요약

**Mom Test** = 사용자에게 “무엇을 만들까?”가 아니라 **“과거에 실제로 무엇이 불편했나?”** 를 묻는 방법.  
산출물은 **진짜 문제 한 문장**이며, 이것이 PRD → spec → TC → 코드 추적의 **맨 위 고리**다.

---

## C2C · ARRR에서의 위치

```
Mom Test (Concept)     ← 이 문서
    ↓  진짜 문제 한 문장
PRD / docs/spec        ← 계약
    ↓
red / green            ← TC · 구현
```

| 단계 | ARRR | Mom Test |
|------|------|----------|
| 요구 발견 | **Ask** (첫 질문) | 과거 사실로 불편 확인 |
| 설계 이후 | Ask (RED) | **반복하지 않음** — PRD·spec 따름 |

---

## 언제 Mom Test를 쓰나

### ✅ 사용

| 상황 | 예 |
|------|-----|
| 새 Epic·기능 논의 | “힌트 기능 넣고 싶어요” |
| 요구가 모호함 | “검증을 더 편하게” |
| PRD/spec에 근거 없음 | 백로그 후보만 있고 AC 미정 |
| 확장 kickoff (⑦) | 기존 진짜 문제와 **같은 불편**인지 확인 |

### ❌ 사용 금지

| 상황 | 대신 |
|------|------|
| TC ID · spec · PRD가 이미 있음 | 문서·TC·`.cursorrules` 따름 |
| RED / GREEN 구현 중 | `pytest.fail` / 최소 구현만 |
| 버그 수정·리팩터 | 실패 TC · 계약 기준 |
| 기술 세부 (에러 코드, fixture 이름) | `docs/spec/` · 기존 패턴 |

---

## 3원칙

1. **솔루션·아이디어 말하지 않기** — “자동 검증 앱”, “`validate_lines` 추가” 등 답을 먼저 제시하지 않는다.
2. **과거·구체·사실만 묻기** — “지난주”, “20분”, “대각선”처럼 검증 가능한 사실만 받는다.
3. **칭찬·미래 의견 무시** — “좋을 것 같아요”, “있으면 편하겠죠”는 수용 기준이 아니다.

### 좋은 답변 vs 나쁜 답변

| 구분 | 예시 | 이유 |
|------|------|------|
| ✅ | “지난주 대각선 빼먹어 20분 날렸다” | 과거·구체·사실 |
| ❌ | “자동 검증 앱 있으면 좋겠다” | 솔루션 |
| ✅ | “행·열만 더하고 대각선은 나중에 확인했다” | 행동·시간 낭비 |
| ❌ | “AI가 채워주면 학습에 도움될 것 같아요” | 미래 의견 |

---

## 산출물 — 진짜 문제 한 문장

Mom Test가 끝나면 **한 문장**만 남긴다 (솔루션·기능명 없음).

> **진짜 문제:** _(과거 사실 기반)_

**이 프로젝트 SSOT** ([`PRD.md`](PRD.md) INV-1):

> **진짜 문제:** 부분 마방진을 채우는 동안 10개 줄의 합(34)을 즉시 판정하고, 틀린 줄을 바로 알 수 없어 확인에 시간이 낭비된다.

### Mom Test 사실 → 설계 연결 (예)

| 확인된 사실 | PRD·설계·TC |
|-------------|-------------|
| 대각선 빼먹어 20분 추가 | 10선 전체 검증 (T2, T3) |
| 행→열→대각선 순 부분 검증 | `validate_lines` 10선 한 번에 |
| 어느 줄인지 모름 | `failed_lines` (R1~D2) |
| 빈칸 vs 틀림 구분 불가 | `incomplete` vs `fail` (T1, T2) |

## 에이전트 행동 규칙

### 1턴 흐름

```
질문 1개
    ↓
사용자 답변
    ↓
사실성 평가 (과거·구체·사실 / 솔루션·미래 여부)
    ↓
추궁 1개 (더 구체화)
    ↓
불편 요약
    ↓
(반복 또는) 진짜 문제 한 문장 제안
```

### 규칙

| # | 규칙 |
|---|------|
| 1 | 한 턴에 **질문 1개**만 |
| 2 | 답변에 **솔루션 제안 금지** |
| 3 | 충분한 사실이 모이면 **진짜 문제 한 문장** 제안 |
| 4 | 확정 전 **`src/` · `tests/` 작성 금지** |
| 5 | 다음 단계: `docs/PRD.md` · `docs/spec/` 갱신 제안 → `spec` 브랜치 |

### 응답 형식 (에이전트)

| 항목 | 내용 |
|------|------|
| **사실성 평가** | ✅/❌ — 과거·구체·사실인지, 솔루션·미래 의견인지 |
| **추궁 1개** | 더 구체적인 **과거 행동** 질문 |
| **불편 요약** | 한 줄로 핵심 불편 |
| **(종료 시) 진짜 문제** | 한 문장 초안 |

## AGENTS.md 초안

아래 블록을 [`AGENTS.md`](../AGENTS.md)에 넣거나, 링크만 두고 이 문서를 SSOT로 쓴다.

```markdown
## 요구 구체화 — Mom Test

**사용 시점:** 사용자 요구가 모호하거나, PRD/spec/TC에 아직 연결되지 않은 **새 기능·Epic**을 논의할 때만.  
**사용 금지:** TC ID·spec·PRD가 이미 있는 작업, RED/GREEN 구현, 버그 수정, 기술 세부(에러 코드·fixture 등).

### 3원칙
1. 솔루션·기능 제안 금지 — "자동 검증 앱", "함수 X 추가" 등 답을 먼저 말하지 않는다.
2. 과거·구체·사실만 묻는다 — "지난번 ~할 때 무엇이 불편했나요?"
3. 미래 의견·칭찬은 근거로 쓰지 않는다 — "있으면 좋겠다"는 수용 기준이 아니다.

### 에이전트 행동
- 한 턴에 **질문 1개**만 한다.
- 답변 후: **사실성 평가** → **추궁 1개** → **불편 요약**.
- 충분하면 **진짜 문제 한 문장**을 제안하고, `docs/PRD.md`·`docs/spec/` 반영을 제안한다.
- 진짜 문제가 확정되기 전에는 `src/`·`tests/`를 작성하지 않는다.

상세: [`docs/mom-test-guide.md`](docs/mom-test-guide.md)
```

## 프롬프트 템플릿

### 사용자 → 에이전트 (기본)

```
MagicSquare Mom Test. 페르소나: 부분 마방진(빈칸 2개) 학습자.
Mom Test 규칙 준수. 질문 1개만. 솔루션 금지.
→ 답변 후: 사실성 평가 + 다음 추궁 1개 + 불편 요약
```

### 1회차 — 시간·맥락

```
MagicSquare Mom Test. 페르소나: 부분 마방진(빈칸 2개) 학습자.
Mom Test 규칙 준수. 질문 1개만. 솔루션 금지.

질문: 지난번 4×4 마방진을 손으로 채울 때, 맞는지 확인하는 과정에서
가장 오래 걸렸거나 다시 시작하게 만든 일이 무엇이었나요?
구체적인 날짜나 상황 하나만 말해 주세요.
```

### 확장 기능 kickoff (예: 범위 밖 숫자)

```
MagicSquare Mom Test. 페르소나: 부분 마방진(빈칸 2개) 학습자.
Mom Test 규칙 준수. 질문 1개만. 솔루션 금지.

새 기능 후보: "1~16 범위 밖 숫자 입력 방지"
질문: 지난번 격자에 0이나 17 같은 값을 넣었을 때,
그걸 언제 어떻게 알았나요? (과거 사실만)
```

회차별 상세 예시·답변 샘플: [`learning-guide.md`](learning-guide.md) §①

## Mom Test 이후 — 다음 단계

| 순서 | 할 일 | 브랜치 |
|------|-------|--------|
| 1 | `docs/PRD.md` INV-1·Epic·C2C 표 갱신 | `green` 또는 `spec` |
| 2 | `docs/spec/<tc>.md` R-G-I-O · 계약 | `spec` |
| 3 | RED 테스트 (`pytest.fail`) | `red` |
| 4 | 최소 구현 | `green` |

Mom Test는 **여기서 끝**이다. 이후는 C2C **판단 항목만 변환** (1 To-Do : 1 TC · RED 먼저).
